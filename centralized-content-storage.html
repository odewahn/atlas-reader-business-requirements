Feature Description

Central Content Storage will provide a destination repository for all transformed content that will be through the Atlas Reader, indexed by Full Content Search, or other content distribution related initiatives.  In addition to raw storage, the CCS will provide a unified API for dealing with content that is keyed to other metadata systems within O'Reilly.  

Key requirements include:

  • Storage support for Atlas HTML builds and EPUB.  
  • Ability to "attach" the storage to instances running various client apps, like the Reader.
  • A JSON API for adding Atlas HTML Builds or EPUB content.
  • A JSON API for adding minimal metadata about the content; we assume that richer metadata will be provided by the oreilly product service
    • A product ID that ties the content to the oreilly product database
    • Basic upload info (username, date, etc)
  • Basic auth over HTTPS 

Non-Requirements include:

  • Sophisticated ACL.  The initial implementation will not include an ACL system for groups, users, organizations, or teams.  It is assumed that client apps will mount the CCS as a root volume and handle ACL at the application level.  
  • Caching.  Caching and other performance related system will be handled at the application layer
  • Direct content serving.  Content will not be served directly; it can only be served by a client app running on an attached server.

 

Success Criteria

To be a success, this project must:

  • Store at least 80% of O'Reilly Frontlist titles by the end of 2014
  • Store at least 80% of the EPUBs from distribution partners
  • Provide a central content repository that can serve the Full Content Search project

Maximum Level of Development Effort

This is an XLarge project -- we'll need to determine resources separately.
